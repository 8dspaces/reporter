Background

ReporterManager – an extendible solution for all your QTP reporting needs.

ReporterManager allows you to easily set up any combination of reporters, set complex rules for each and every reporter, and immediately use them in your scripts. After setting the reporters once, using the mechanism boils down to one simple command: oReporterManager.Report.

You can choose from (so far) 8 reporting engines - User timed popup message, XML, Excel, Windows event log, DB reporter, Styled Text, Screen Capture, and the native QTP reporter.

You can and apply as many Regular-Expression and Object-State filters on them as you'd like.

Aside from the current engines and filters, ReporterManager can be easily extended to include countless more reporting and filtering techniques.

Just use the included class template to easily create engines and filters that suite your needs.

To illustrate ReporterManager's strength - with ReporterManager you can:

1. Set up a text backup log (in case QTP will crash and the results will be lost forever), styled for convenient reading.

2. Set up a special text Error log (which only includes failed steps), for quick skimming, and save pictures of all the errors.

3. Set up a fancy Excel log, which only logs performance timers.

And you can do all that with just 10 code lines.

 

Sounds good? Then let's see how exactly this is done.

I recommend on skimming through the entire read-me, but if you can't wait, jump to Wrapping it all together, and see the actual code used.

 

 

Parameters Syntax

Throughout ReporterManager different classes and object, there's a standard way for specifying settings parameters.

The parameters are always sent as one structured string formatted as "Param1>Field1|Param2>Field2…"

e.g. "File>C:\log.txt|ClearExisting>True"

 

Notice that parameter names and values are case sensitive (e.g. "ClearExisting>True" isn't like "clearexisting>True").

 

If you need the characters ' | ', ' > ' or ' " ' to appear as actual data, double them. i.e. write >> instead of >, etc.

e.g. "Pattern>A||b" means there's just one parameter – "Pattern", with value = "a|b".

            Similarly, "BodyTemplate>vbcrlf & "" String "" ", means that parameter BodyTemplate has value = vbcrlf & "String"

Structure

ReporterManager consists of 3 components blocks:

1. The manager class object.

2. The reporter-engines classes.

3. The filter classes.

1. The manager class object Interfaces

This is the heart of the ReporterManager mechanism. The manager object holds all the different reporters and filters, manages them and uses them to report the relevant events.

 

Object Model

The object is accessible via the global single-tone variable – oReporterManager.

oReporterManager (ReporterManager Class)

   \-----oReporters (Dictionary) – All the registered reporters

                        \----- Reporter (Dictionary) – Represents an engine with filters

                                                |----- Engine (Engine Class) – The reporter's engine

                                                \----- Filters Collection (Dictionary) – All the filters registered to this engine

                                                                        \-----Filter (Filter Class) – A filter mechanism

 

Methods And Properties

.Version

            Goal – A simple property, returning the current version of the class.

 

.StartEngine (sType, sName, sSettings)

Goal: Starts engine sName of type sType.

In Params:

sType - The Class of the engine (e.g. "Excel")

sName - The name of the relevant reporter (e.g. "My Log")

sSettings - The settings to send the engine as "Field1>Value1|Field2>Value2". Can be empty.

Depends on the specific engine (e.g. "ClearExisting>True|File>c:\log.txt")  

Out Params:

Boolean - Success/Failure

 

            .StopEngine (sName)

                        Goal: Stops engine sName.

In Params:

sName - The name of the relevant reporter (e.g. "My Log")

 

            .Report (sStatus, sStepName,sExpected,sActual, sDetails)

                        Goal: Reports an event to all possible reporters.

In Params:

sStatus - The status of the event : "Fail", "Pass", "Warning" or custom

sStepName - The name of the event (e.g. "Validate Field Value")

sExpected - The expected result (e.g. "Value = Something")

sActual - The actual result (e.g. "Value = Something Else")

sDetauls - Possile details about the event

 

            .ReportToLog(sReporterName, sStatus, sStepName,sExpected,sActual, sDetails)

                        Goal: Just like .Report, only it reports just to sReporterName.

 

.AddFilter(sFilterType, sReporterNames, sFilterSettings)

                        Goal: Adds a filter to a specific log.

In Params:

sReporterNames - The names of the relevant reporter as "Reporter1>Reporter2>…"

                                                Please note – if the reporter name contains that character ' > ', double it.

E.g. .AddFilter("FilterType", "Name1>Reporter>>Name2>", "FilerSettings")

                                                The command sets the filter for 2 engines: "Name1" and "Reporter>Name2".

sFilterType - The class of the filter (e,g, "RegEx")

sFilterSettings - The settings to send the filter as "Field1>Value1|Field2>Value2". Can be empty

                                    Depends on the specific filter (e.g. "Incldue>True|Pattern>.*Error.*")           

Out Params:

Boolean - Success/Failure

 

So here's a quick example as to using oReporterManager:

Call oReporterManager.StartEngine("Text", "Log", "") 'Start Text Log with default settings

Call oReporterManager.Report ("Pass", "Step 1" ,"Window Should Open" ,"All OK", "No Details")  'Will log the event into the file

 

2. The reporter-engines classes

The real power of the ReporterManager lies within the options of the different report-engines and filters. Let's get to know them.

The engines that come with this version are: Excel, Windows event log, Text, User on-screen messages, DB reporter, Screen Capture and the native QTP report.

This read-me only specifies the settings each engine can receive. For a list of methods, properties and inner procedures, see Further documentation.

 

XML – Reports to an external XML file. A default QTPLog.XLS file is supplied with the project to enable instant HTML report.

            This engine can receive the following settings:

                        File = String - the log file path and name. (e.g. "C:\LogFile.XML"). Default = "C:\QTPLog.XML"

                        ClearExisting = Boolean - Should we erase the existing log (True = Erase). Default = True

                                                                If the reporter can't parse the existing XML file, it will be overwritten.

                        AutoSave = Integer - Save data to file every X minutes. e.g. 2. Default = 0

                                                                Higher value = better performance. Lower value = better robustness.

                        StyleSheet = String - the XSL stylesheet file path and name. (e.g. "C:\QTPLog.XSL"). Default = "C:\QTPLog.XSL"

                                                                If value = "", no stylesheet instruction will be entered.

                                                                A default QTPLog.XLS file is supplied with the project to enable instant HTML report.

 

            e.g. : oReporterManager.StartEngine("XML", "MyLog", "File>C:\QTP\Log.XML|ClearExisting>False" )

            This will write the log to C:\QTP\Log.XML, appending the XML if the file already exists.

               

                Here's the XML structure the reporter creates, in case you'd want to write your own XSL template wrapper:

                                Root Name= QTPLogs

                                Each time the engine is started, a new QTPLog element is created. Attributes = TestName, Start, End, Status

                                                Each QTPLog contains several Iteration elements. Attributes = Number, Start, Status

                                                                Each Iteration contains several Step elements

                                                                                Each Step contain the following sub-elements (all with CData values):

                                                                                                Status, StepName, Expected, Actual, Details, Time

 

Excel – This engine is mainly a wrapper on Sameer Girolkar's ResultToXL. It will create a stylish excel log.

            This engine can receive the following settings:

                        File = String - the log file path and name. (e.g. "C:\LogFile.Txt"). Default = "C:\QTPLog.xls"

                        ClearExisting = Boolean - Should we erase the existing log (True = Erase). Default = True

 

            e.g. : oReporterManager.StartEngine("Excel", "MyLog", "File>C:\QTP\Log.xls|ClearExisting>False" )

            This will write the log to C:\QTP\Log.xls, appending the text if the file already exists.

 

QTP – This is just the regular QTP reporter.ReportEvet, wrapped to work with ReporterManager.

            Please Note – Only ONE QTP engine can be active at any given time.

            This engine can receive the following settings:

                        Filter = Integer - 0 / 1 / 2 / 3. Sets the filtering on the native QTP results file (see Reporter.Filter help)

                                    Default = 1 ( rfEnableErrorsAndWarnings = Only warning and fail are logged.)

 

            e.g. : oReporterManager.StartEngine("QTP", "MyLog", "Filter>3" )

            This will block any non-error event QTP would usually log (e.g. Button.Click etc.)

 

User – Pops up a message for the user. The message disappears after a few seconds.

            This engine can receive the following settings:

                        Timer = Integer – The time until the message disappears. Default = 3.

 

            e.g. : oReporterManager.StartEngine("User", "MyLog", "Timer>10" )

                        This will show the report for 10 seconds before continuing the script

 

ScreenCapture – Captures the current screen and saves it to file.

            This engine can receive the following settings:

                        Path = String – The path to save the pictures in. Default = "C:\".

                        Prefix = String – a prefix for the file names. Default = "QTP Capture"

                       

                        File names always end with the event's time-stamp.

                        If a file already exists, it is saved with the suffix "_1.png".

 

            e.g. : oReporterManager.StartEngine("ScreenCapture", "Error Screens", "Prefix>Error -|Path>C:\Errors\" )

 

WinLog – Reports to the windows event log (requires WinXP/ 2000/ Vista).

            This engine can receive the following settings:

ShowAs – String - The source column. Default = Name of test.

ShowAs must be computer-unique, so better not use the AUT name

e.g. ShowAs = "MsiInstaller" will fail to create the engine, because that's an existing system name

ClearExisting – Boolean. Should we erase all entries in the log? Default = True

                                    Computer – String. The IP / Network name of the computer to log the report. Default = "." (local)

                                                                        Make sure the current user has log writing privileges to the given computer.

LogName – String. The name of the log. Defaulted to "QTP Log"

DO NOT USE THE LogName SETTING UNLESS YOU KNOW WHAT YOU'RE DOING.

If ShowAs was previously used in a different log-name, you would have to reboot for the changes to take effect 

                e.g.: oReporterManager.StartEngine("WinLog", "Error ", "ShowAs>Siebel Error" )

 

DB – Reports to a DB.

                Please Note –    The DB must already exist. If you do not set CreateNew = True, the table must have the fields specified.

                                                If a field name is set to "", the field won't be created in a new table, and won't be logged.

            This engine can receive the following settings:

                                                ConnectionString - String - The database connectionstring text.

                                                                                                Defaulted to "DBQ=C:\QTP Log.mdb;UID=admin;Driver={Microsoft Access Driver (*.mdb)}"

                                TableName – String – The DB Table Name. Defaulter to "Log"

                                                                                Please notice, any string will be used as uppercase. So "log" = "LOG"

                                CreateNew – Boolean - Should we Create a new table (deleting exsiting). True = Erase. Defaulted to True.

                                                                                USE WITH EXTREME CAUTION. The table will be completely dropped.

                                                StepNameField - String - The DB fieldname for StepName. Defaulted to "StepName"

                                                StatusField String - The DB fieldname for Status. Default = "Status"

                                                ExpectedField - String - The DB fieldname for Expected Result. Default = "Expected"

                                                ActualField - String - The DB fieldname for Actual Result. Default = "Actual"

                                                DetailsField - String - The DB fieldname for Details. Default = "Details"

                                                TimeField - String - The DB fieldname for Details. Default = "EventTime"

 

                e.g.: oReporterManager.StartEngine("DB", "Error ", " TableName >Error Log|StatusField>EventStatus|CreateNew>True" )

                This will drop the "Error Log" table in c:\QTP Log.mdb (if it exists) & create a new table with status column = "EVENTSTATUS"

 

Text – Your everyday write-to-text-file log, but with possible template formatting.

            This engine can receive the following settings:

                        File = String - the log file path and name. (e.g. "C:\LogFile.Txt"). Default = "C:\QTPLog.xls"

                        ClearExisting = True/False - Should we erase the existing log. True = Erase. Default = True

                        BodyTemplate = String. The template for reporting the actual event

NewTestCaseTemplate = String. Template for the title when switching actions / iterations.

 

                        Explaining Templates:

                                    Both templates have to be a valid REPRESENTATION of a string (as opposed to an actual string).

                                                Usually this would mean doubling all the double quotes, and wrapping the whole expression with double-quotes

                                                So this is a valid REPRESENTATION: " vbcrlf & ""Hello""  ".

This is not: vbcrlf & "Hello". Nor " vbcrlf & "Hello" " (which isn't even a valid string)

 

                                    Within the template string, you can specific the following tags:

                                                %Status%                   = Will be replace with the event status

                                                %Details%                  = Will be replace with the event Detai

                                                %StepName%            = Will be replace with the event Step Name

                                                %Expected%              = Will be replace with the event expected result

                                                %Actual%                    = Will be replace with the event actual result

                                                %TestCaseName%    = Will be replaced with the event Test Case Name (action name + iteration)

                                                %Time%                     = Will be replaced with the event time stamp

 

                                    So the template " "" Status = %Status% "" & vbcrlf & "" Step= %StepName%  "" will produce (for example):                         

                                                Status = Fail

                                                Step = Input field

 

                        A more complex example:

"vbcrlf & ""%StepName%, %Status%"" & vbcrlf & String(20, ""-"") & vbcrlf &"" Expected :  %Expected%"" & vbcrlf & ""Actual : %Actual% "" & vbcrlf & ""Details: %Details% "" "

 

                                                This will produce a log like:

Step 1          Pass

--------------------

Expected Result:  Window Should Open

Actual Result: All OK

Details: No Details

                        One last example:

A very common use is to create a valid .csv file that can later be used for Excel statistical analysis.

Use these two templates parameters to produce a minimal but powerful .CSV log:

"BodyTemplate>""%Status%, %StepName%, %Expected%, %Actual%, %Details%"" & vbcrlf"

"NewTestCaseTemplate>""General, Switched TestCase, , , %TestCaseName%"" & vbcrlf"

 

3. The filter classes

Filters are used to funnel different types of data to different logs.

You would have little use for 30 logs which all have the same data, over and over again. But, it could be very useful to have a special errors log, a dedicated performance log, etc.

Before ReporterManager, you would have had to either use different report commands within your code, or use messy logic with-in your central report function. Filters take the load off your shoulders, while giving you very extensive conditional complexity.

When you use the oReporterManager.Report command, it loops through all the reporters, and tries to log the event through them. Filters screen OUT events from certain reporters.

So for example, you can set up an error log by setting up a new reporter, and applying a Regular Expression filter on it, which only allows events with Status=Fail. You can apply as many filters as you'd like on a reporter, thus achieving almost unlimited conditional power.

Multiple filters are always connected via AND conditional (i.e., enough for one to fail, and the event won't be logged)

 

The filters that come with this version of ReporterManager are RegEx regular expression filter, and Object Property value filter:

This read-me only specifies the settings each filter can receive. For a list of methods, properties and inner procedures, see Further documentation.

 

RegEx – A powerful regular expression filter, which allows you to set textual filtering on any part of the reported event.

            This Filter can receive the following settings:

                        Include               = Boolean. Is the filter defined positively (True) or negatively (False). Default = True

                                                      i.e. is Finding the RegEx considered a success (True), or NOT Finding it (False)

                        WhatToSearch = "Value1, Value2,...". ValueX = one of Status/StepName/Expected/Actual/Details

                                                In what parts of the event to report should we search for the RegEx

                                                Default = "Status, StepName, Expected, Actual, Details" (all parts)

                        IgnoreCase     =    Boolean. Should we ignore string case in our searches? Default = True

                        Pattern            =   String. The RegEx pattern to seach (e.g. ".*Error.*")

Remember to use '||' instead of '|' if your pattern includes this char.

e.g. use "Pattern>Fail||Warning|IgnoreCase>True"

 

                        Notice that when Include=True, even one hit will count as a filter match, and will be authorized.

                                    e.g. if WhatToSearch = "Status, StepName", a hit in Status (alone) will pass the filter.

                        When Include = False, even one hit will count as a filter mismatch, and will be blocked.

                                    e.g. if WhatToSearch = "Status, StepName", a hit in Status (alone) will fail the filter.

 

e.g. : oReporterManager.AddFilter("RegEx", "Log", "Include>False|WhatToSearch>Status|Pattern>Warning")

                                                This will filter OUT all events with status = Warning (because Include = False)

                                                All other events will pass the filter, and will be logged

ObjectProperty – Checks values of any object property / VBScript statement (e.g. if a window exists).

                                    This is most handy when you suspect something might go wrong when a trigger event occurs.

                                    So maybe you'd like to log performance only when a certain app. Window is opened, etc.

Please Note – When using this filter, make sure your QTP object sync timeout is LOW (i.e. 1 / 2 sec.)

(Otherwise the script execution time will increase significantly)

 

            This Filter can receive the following settings:

                        ObjectProperty            = String. A REPRESENTATION of an object property (QTP or other). Default = ""

e.g. " VBWindow(""vbname:=Something"").Exist "

USE ONLY DP, not OR objects.

You can use any VBScript command that can be properly represented.

e.g "Trim(VBWindow(""vbname:=Something"").Object.Date)"

 

                        Value                           = String. The expected property of the value. Default = ""

                        Condition                     = Boolean.       True -> a match is considered a success.

                                                                                    False -> a match is considered a failture. Default = True

 

                                    e.g. "ObjectProperty>Window(""name:=x"").getROProperty(""text"")|Value>Notepad|Condition>False"

                                    This will FAIL if the window's text = "Notepad", and will PASS if it's anything else

 

Wrapping it all together

So, let's go back to the claim I made earlier. I said that with ReporterManager you can:

1. Set up a text backup log (in case QTP will crash and the results will be lost forever), styled for convenient reading.

2. Set up a special text Error log (which only includes failed steps), styled for quick skimming.

3. Set up a fancy Excel log, which only logs performance timers.

And you can do all that with 10 code lines. So, here are 8 code lines that do the trick (I kinda cheated – I didn't count the comments :) )

 

'Task 1 – Set up a backup text log. No filtering – everything is logged

   Call oReporterManager.StartEngine("Text", "Backup", "File>C:\QTPLog.txt")   

 

'Task2 – Set up an error text log which only reports Fail/Warning events. Set up a screen captures for errors

   Call oReporterManager.StartEngine("Text", "Errors", "File>C:\ Errors.txt|BodyTemplate>""%Status%:   %StepName%, %Expected%, %Actual%""")

   Call oReporterManager.AddFilter("RegEx", "Errors", "Pattern>Fail|WhatToSearch>Status")

   Call oReporterManager.StartEngine("ScreenCapture", "Error Screens", "Path>C:\|Prefix>Error - ")

   Call oReporterManager.AddFilter("RegEx", "Errors", "Pattern>Fail|WhatToSearch>Status")

 

 'Task3 – Set up an excel log, which only logs events referring to opening a window, with status = Not Warning

   Call oReporterManager.StartEngine("Excel", "Top View", "File>C:\TopView.xls")

   Call oReporterManager.AddFilter("RegEx", "Top View", "Pattern>.*Window.*Open.*|WhatToSearch>Expected,Details")

   Call oReporterManager.AddFilter("RegEx", "Top View", "Include>False|Pattern>Warning|WhatToSearch>Status")

 

And that's just about it. Pretty cool, ha?

If you'd like to test it, just throw in these code lines, which do the actual reporting:

 

  'Will go to Backup and TopView

 Call oReporterManager.Report ("Pass", "Step 1" ,"Window Should Open" ,"All OK", "No Details")

 'Will go to Backup

 Call oReporterManager.Report ("Warning", "Step 2" ,"Window Should Open" ,"It Didn't", "No Details")

 'Will go to Backup, Errors and TopView, and will save a screen capture

 Call oReporterManager.Report ("Fail", "Step 3" ,"Window Should Open" ,"App. Crashed", "Bla Bla Bla")

 

Enjoy :)

Extending ReporterManager

ReporterManager is structured so that it could be easily extended.

Each filter and reporting engine is encapsulated in its own class, so adding new ones boils down to writing distinct, separated chunks of new code.

Once a new reporter-class or filter-class is created, ReporterManager could work with them immediately, with zero changes to existing code.

Aside from ReporterManager.VBS, Filters.VBS and Engines.VBS, you'll find "Class Templeates.VBS", which shows what functions and procedures the new classes must implement.

Just copy the template to a new file, and start filling it up with content. And of course, don't forget to let me know, so I can spread the word, and include your work with the next ReporterManager release.

          Upgrading extensions

If you've upgraded a previous version of ReporterManager, Please look in the detailed change-log (available in Changelog.htm), and in the new engine and filter class templates, to understand the changes you must implement to your existing code.
