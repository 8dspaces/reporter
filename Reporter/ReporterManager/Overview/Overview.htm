Overview

ReporterManager ñ an extendible solution for all your QTP reporting needs.

ReporterManager allows you to easily set up any combination of reporters, set complex rules for each and every reporter, and immediately use them in your scripts. After setting the reporters once, using the mechanism boils down to one simple command: oReporterManager.Report.

You can choose from (so far) 8 reporting engines - User timed popup message, XML, Excel, Windows event log, DB reporter, Styled Text, Screen Capture, and the native QTP reporter.

You can and apply as many Regular-Expression and Object-State filters on them as you'd like.

Aside from the current engines and filters, ReporterManager can be easily extended to include countless more reporting and filtering techniques.

Just use the included class template to easily create engines and filters that suite your needs.

To illustrate ReporterManager's strength - with ReporterManager you can:

1. Set up a text backup log (in case QTP will crash and the results will be lost forever), styled for convenient reading.

2. Set up a special text Error log (which only includes failed steps), for quick skimming, and save pictures of all the errors.

3. Set up a fancy Excel log, which only logs performance timers.

And you can do all that with just 10 code lines.

 

Sounds good? Then let's see how exactly this is done.

Parameters Syntax

Throughout ReporterManager different classes and object, there's a standard way for specifying settings parameters.

The parameters are always sent as one structured string formatted as "Param1>Field1|Param2>Field2Ö"

e.g. "File>C:\log.txt|ClearExisting>True"

 

Notice that parameter names and values are case sensitive (e.g. "ClearExisting>True" isn't like "clearexisting>True").

 

If you need the characters ' | ', ' > ' or ' " ' to appear as actual data, double them. i.e. write >> instead of >, etc.

e.g. "Pattern>A||b" means there's just one parameter ñ "Pattern", with value = "a|b".

            Similarly, "BodyTemplate>vbcrlf & "" String "" ", means that parameter BodyTemplate has value = vbcrlf & "String"

 

Extending ReporterManager

ReporterManager is structured so that it could be easily extended.

Each filter and reporting engine is encapsulated in its own class, so adding new ones boils down to writing distinct, separated chunks of new code.

Once a new reporter-class or filter-class is created, ReporterManager could work with them immediately, with zero changes to existing code.

Aside from ReporterManager.VBS, Filters.VBS and Engines.VBS, you'll find "Class Templeates.VBS", which shows what functions and procedures the new classes must implement.

Just copy the template to a new file, and start filling it up with content. And of course, don't forget to let me know, so I can spread the word, and include your work with the next ReporterManager release.

          Upgrading extensions

If you've upgraded a previous version of ReporterManager, Please look in the detailed change-log (available in Changelog.htm), and in the new engine and filter class templates, to understand the changes you must implement to your existing code.
